# Char PP Prj (基于 PaddleVideo 的乒乓球动作识别) 项目章程

## 核心原则

### I. 规范与计划优先 (Spec-Driven, Plan-Gated)

任何新功能、模型改造或流程变更, 在进入实现之前**必须**有对应的 `spec.md` 与 `plan.md`,
且必须通过本章程的"章程检查"门控. 实现产生的代码与配置必须可追溯到至少一条功能需求 (FR-xxx) 或用户故事.

**理由**: 本项目同时涉及上游框架复现与业务建模, 没有规范驱动很容易出现"调通了某个示例就开始堆代码"
的失控. 规范优先确保每一行代码都服务于已经被审视过的需求.

### II. 可复现实验 (Reproducible Experiments)

每一次训练、评估或对外报告的指标, **必须**满足以下条件之一才能被视作"有效实验":

- 使用纳入版本管理的配置文件;
- 显式设置随机种子, 并在日志中记录;
- 在日志或实验目录中记录: 代码 commit、配置哈希、数据集划分版本、起止时间、最终指标;
- 在硬件相同时, 同一份配置 + 数据 + 种子重跑两次, 主指标差异 ≤ ±2 个百分点 (与 spec.md SC-004 对齐).

不满足以上要求的"一次性结果"**不得**写入文档、报告或决策依据.

**理由**: 视频识别的指标受随机性、数据划分、采样策略影响极大, 没有可复现性, 项目讨论会迅速退化为
"我那次跑出来是 78%". 这是不可协商的工程纪律.

### III. 配置驱动, 拒绝硬编码 (Config-Driven, No Hard-Coding)

数据路径、动作类别集合、模型结构、采样窗口、批大小、学习率、训练轮数、保存路径等
**所有**与实验相关的可调参数, **必须**通过配置文件驱动. 源码中**不允许**出现:

- 硬编码的本地绝对路径;
- 硬编码的类别名称或类别数量;
- 硬编码的训练超参数.

针对"调试中的临时硬编码", 允许在不超过一个工作日内存在, 但**不得**进入主分支.

**理由**: 与原则 II 紧密耦合——只有配置驱动, 实验才能被准确复现; 同时也是 spec.md FR-008/FR-017
的直接落地.

### IV. 数据完整性 (Data Integrity & No Leakage)

数据准备脚本与训练流程**必须**保证以下不可妥协的事实:

- 训练集、验证集、测试集在**视频源**层面互不重叠 (一个原始视频被切成的所有片段必须落在同一个划分中);
- 数据划分一旦发布, 划分文件本身被纳入版本管理, 任何重新划分都视为新的实验;
- 输入异常 (损坏视频、过短片段、不支持格式) **必须**被捕获并记录, **不得**让整体流程崩溃;
- 测试集**绝不**用于早停、超参选择或模型选择.

**理由**: 视频片段层级的数据泄漏是评估指标失真的最常见原因; 测试集污染会使所有后续决策都建立在
错误的指标之上. 这是项目能否被信任的底线.

### V. 评估纪律 (Evaluation Discipline)

每一个对外披露的模型版本, **必须**至少给出以下指标:

- Top-1 准确率;
- Top-5 准确率;
- 每类指标 (混淆矩阵或分类报告);
- 测试集大小与各类样本数量.

仅给出"整体准确率"的报告**不得**作为模型上线、模型选择或方案选型的依据. 当类别极不平衡时,
**必须**额外汇报宏平均 (macro-average) 指标.

**理由**: 乒乓球动作类别天然长尾 (例如发球、正手攻样本远多于削球), 仅看整体准确率会系统性高估
小类性能; 这条原则强制让每次评估暴露真实情况.

### VI. 上游兼容与最小侵入 (Upstream Compatibility, Minimal Intrusion)

复现 PaddleVideo 不等于把上游代码搬进来重写. 集成上游时**必须**遵守:

- 优先以子模块、克隆、或可重新拉取的方式接入上游, 不直接把上游代码与本仓库代码混合;
- 业务相关的扩展 (乒乓球数据、配置、脚本) 放在专属目录, 不污染上游目录;
- 必须修改上游源码时, 用补丁 (patch) 或 fork 分支记录修改, 并在文档中说明原因;
- 不得为了"看起来更整洁"而删除上游许可证、致谢或原始 README.

**理由**: 这是合规要求, 也是工程可持续性的要求——上游升级时, 我们能干净地拉取新版本而不是陷入
合并地狱; 同时也尊重 PaddleVideo 项目的开源许可证.

### VII. 端到端可运行 (Runnable End-to-End)

仓库的 main 分支**必须**始终保持以下状态: 给定一份合规的乒乓球数据集和一份配置文件, 用户可以通过
不超过 5 条命令完成 "环境验证 → 数据准备 → 训练 → 评估 → 推理 → 可视化" 全流程. 任何会破坏这一性
质的合并**必须**在合并说明中明确指出, 并在同一 PR 内修复.

**理由**: 项目的最终交付是"能用"而不是"代码漂亮". 端到端可运行是判断项目健康度的唯一外部信号,
也直接对应 spec.md 中 FR-019 与 SC-001.

### VIII. 隔离的 Python 环境 (Isolated Python Environment)

本项目**必须**使用项目独立的 Python 环境, **严禁**直接使用系统 Python 或与其他项目共享的全局环境.
不可协商的硬性约束如下:

- **唯一目标版本**: Python 3.11 (3.11.x 任意补丁号皆可); 不允许在主分支引入 3.10 或 3.12 的兼容代码,
  也不允许同时维护多个 Python 版本的并行实现;
- **隔离方式**: 必须通过 `venv` / `conda` / `uv` / `virtualenv` 之一在项目目录内 (或受项目脚本管理的
  独立路径) 创建环境; 该环境**不得**与系统 Python 共享 site-packages;
- **入口约束**: 所有可执行脚本、训练入口、推理入口、CI 任务**必须**显式调用项目环境的解释器
  (例如通过 `./venv/bin/python` 或激活脚本), **不得**在 shebang 或文档中出现 `#!/usr/bin/env python`
  这种可能解析到系统 Python 的写法;
- **依赖锁定**: 必须存在一份能在 Python 3.11 上一次性还原项目环境的依赖清单 (如 `requirements.txt`
  + `requirements-lock.txt`, 或 `environment.yml`, 或 `pyproject.toml` + lock 文件); 锁文件**必须**入库;
- **上游适配**: 从上游 PaddleVideo 仓库拉取的代码或环境清单, 如与 Python 3.11 不兼容
  (例如 `print` 语法、被弃用的 `collections.Mapping`、`numpy<1.24` 与 Python 3.11 的 ABI 冲突等),
  **必须**在本项目的"上游适配层"或补丁中修复, 并在文档中记录原因; **不得**通过降级到 3.10 或更早版本
  绕过问题;
- **环境验证**: 端到端 quickstart 中**必须**包含一条可执行的环境自检命令, 在版本不符或解释器路径
  指向系统 Python 时立即失败并给出清晰的修复指引.

**理由**: 视频识别项目对底层依赖 (PaddlePaddle、CUDA 工具链、OpenCV、ffmpeg 绑定) 极其敏感, 一旦
环境被污染, 复现实验和 bug 排查的成本会指数级上升; 同时, 与系统 Python 共享会在多人协作和服务器部署
时引入难以预期的破坏. 锁定 Python 3.11 是为了让"上游适配工作"只发生一次, 不在不同 Python 版本上反复
踩坑.

## 工程约束与质量门

- **运行环境**: 默认目标为 Linux + 单 GPU; 必须保留 CPU 模式作为环境验证回退路径 (spec.md 边界情况).
- **Python 运行时**: Python 3.11 + 项目隔离环境 (原则 VIII); 不允许直接调用系统 Python 解释器.
- **依赖管理**:
  - 所有 Python 依赖必须固定主版本, 并提供可一键还原的锁文件;
  - 与 PaddlePaddle/CUDA 的兼容矩阵必须在文档中维护;
  - 上游 PaddleVideo 的 `requirements.txt` 不得直接生效, **必须**经过 Python 3.11 适配层
    (例如本项目维护的 `requirements/upstream-py311.txt` 或等价的 patch) 才能纳入安装流程.
- **目录约定**:
  - `specs/` —— 规范、计划、清单 (由 speckit 管理, 不得手工随意改动);
  - `configs/` —— 所有训练/评估/推理配置;
  - `data/` —— 数据集 (受 .gitignore 管控, 不入库);
  - `experiments/` —— 实验输出 (日志、checkpoint、指标);
  - 上游 PaddleVideo 集成放在专属子目录, 与上述目录隔离 (原则 VI);
  - 项目隔离 Python 环境的目录 (如 `.venv/`, `venv/`, `env/`) **必须**在 `.gitignore` 中, 不得入库.
- **质量门 (在 plan/tasks 阶段强制核对)**:
  1. 配置驱动: 不得出现硬编码业务参数;
  2. 数据划分文件入库;
  3. 实验目录包含 commit / 配置 / 种子 / 指标四元组;
  4. 端到端 quickstart 命令在最近一次合并时已验证;
  5. 任何评估只在留出测试集上进行一次, 严禁反复挑选;
  6. 环境自检通过: 解释器为项目隔离环境且版本为 Python 3.11 (原则 VIII).

## 开发工作流

- **新增功能**: `/speckit.specify` → `/speckit.clarify` (可选) → `/speckit.plan` → `/speckit.tasks`
  → 实施 → 评估 → 合并. 跳步**必须**在 PR 描述中说明理由.
- **章程检查**: `/speckit.plan` 阶段必须对照本章程逐条检查; 任何违规进入"复杂度跟踪"表格,
  并提供"为什么需要 / 为什么更简单方案不够"的说明.
- **实验合入**: 新指标进入文档前, 必须附上对应的实验目录引用 (commit, 配置文件, 日志路径);
  无引用的指标不得作为决策依据 (原则 II).
- **代码评审**: PR 必须明确列出受影响的原则编号 (例如 "对应 III, IV, VIII"), 评审者据此核对.
- **回滚**: 一旦发现已合并代码违反原则 IV (数据泄漏)、原则 V (评估失真) 或原则 VIII
  (污染系统 Python / 偏离 3.11 目标版本), **必须**优先回滚或修复, 不得以"先发后修"心态处理.

## 治理

- 本章程优先于所有非正式的实践和惯例. 与本章程冲突的代码、配置、文档**必须**以本章程为准修改.
- 修订流程: 提议者提交一份说明文件描述修订理由, 影响哪些原则与模板, 并据语义版本规则给出版本递增建议
  (MAJOR: 删除/不兼容地重定义原则; MINOR: 新增原则或实质性扩展; PATCH: 措辞、澄清). 修订**必须**
  同步更新模板对齐状态 (本文件顶部的同步影响报告), 不允许出现"章程已改但模板未对齐"的中间状态.
- 复杂度证明: 任何引入额外复杂度 (新框架、新存储、新流程) 的决定都**必须**在 plan.md 的"复杂度跟踪"
  表格中说明, 否则不予合并.
- 合规审查: 每次重大合并 (新模型、新数据划分、上游升级、依赖升级) 后, 必须由作者自检本章程全部
  核心原则; 评审者在 PR 评论中确认原则合规性方可合入.
