## 命令

因为 BMN 20-epoch 训练仍在同一 GPU 跑 (PID 3571466, 已 16h+), 需要降低
extractor 显存占用避免 OOM. 临时 yaml override:

```bash
# 准备低 GPU 占用的 extractor 配置 (batch_size=4, gpu_mem_mb=1500)
cp configs/models/pp_tsm_extractor.yaml /tmp/pp_tsm_extractor_safe.yaml
sed -i 's/batch_size: 32/batch_size: 4/' /tmp/pp_tsm_extractor_safe.yaml
# 再手动加 runtime.gpu_mem_mb: 1500

# 端到端推理
.venv/bin/pp infer-rawvideo \
    --input data/raw/test_videos/06_zhengshou_kuaidai.mp4 \
    --bmn-checkpoint experiments/<run>/BMN_epoch_00013.pdparams \
    --output-dir /tmp/cos_real_demo \
    --no-visualize \
    --threshold 0.3 \
    --min-duration 0.5 \
    --extractor-config /tmp/pp_tsm_extractor_safe.yaml \
    --allow-dirty
```

### 关键洞察

**模型学到了什么 (基于 13 epoch 未完全训练的 ckpt)**:
- ✅ **高置信区间集中在视频中后段** (3:00+, 4:00+), 这与教学视频"先讲解后示范"的结构吻合
- ✅ **第 1 个高分区间在 0:12** — 也就是视频开头后经过 ~12 秒的简短介绍后, 教练做了第一次动作示范

- ⚠️ **label_name=unknown**: BMN 只做 temporal localization (何时发生动作), **不做动作分类**. 上游 FootballAction 在 BMN 后接 LSTM head 做分类, 本项目未接 (out-of-scope, 留待 003 feature 候选)

## 已知限制

### GPU 共享下的性能损失

- **理想**: 独占 T4 + batch_size=32 → 4:39 视频约 70 秒端到端 (~4x 实时)
- **本次**: 与训练共享 GPU + batch_size=4 + gpu_mem_mb=1500 → 3:53 端到端 (~0.83x 实时)
- **原因**: batch_size=4 时 PP-TSM forward 吞吐掉到 32 fps (vs 独占时 ~80 fps); `memory_optim` 开销增加

### `pp infer-rawvideo` 缺少 `--batch-size` 参数

实测时发现 `pp infer-rawvideo --help` 只有 `--extractor-config`, 没有直接的
`--batch-size` / `--gpu-mem-mb` 便捷参数. 需要用户手动复制 yaml 改参数.

**后续改进** (非 blocker): 给 `pp infer-rawvideo` 加 `--batch-size` 与 `--gpu-mem-mb`
直接覆盖, 避免 yaml 编辑繁琐. 列入 003 feature 或 v0.3.1 patch-release.

### `label_name=unknown`

BMN 本身不输出类别标签, 这是**架构正确行为**. 不是 bug, 但对业务用户而言
"只知道什么时候有动作, 不知道是哪种动作"的产品价值有限. 真正的用户价值需要
003 feature 接 LSTM / Transformer head 做分类.
